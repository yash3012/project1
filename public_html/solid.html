SOLID Object Oriented Design

SOLID is a set of object oriented design principles aimed at making code more maintainable and flexible.

SOLID is an acronym that stands for the following:

  • Single responsibility principle
  • Open/closed principle
  • Liskov substitution principle
  • Interface segregation principle
  • Dependency inversion principle

Single Responsibility Principle :

This states that a class should have a single responsibility, but more than that, a class should only have one reason to change. Here the calculator.py class has only one task to do. That is to call the addition, subtraction, multiplication and division method. The actual calculations are being performed in their respective classes which is being inherited by the calculator.py class.

Open/Closed Principle :

In the open/closed principle classes should be open for extension, but closed for modification. Essentially meaning that classes should be extended to change functionality, rather than being altered into something else. In our case, the addition function when initially created only added 2 numbers. after that, we refactored the method, and added such functionality that it can add in more than two numbers. However, we cannot make this method perform another type of calculation such as multiplication. For that we need to change that calculation's respective class method.

Liskov Substitution Principle :

This states that objects should be replaceable by their subtypes without altering how the program works. In other words, derived classes must be substitutable for their base classes without causing errors. In our case, we can see in the above picture that Addition.py class extends Calculation.py class. It is able to perform all the functions that Calculation.py class can perform and on top of that, it can also perform the addition function as well.

Interface Segregation Principle :

This states that many client-specific interfaces are better than one general-purpose interface. In other words, classes should not be forced to implement interfaces they do not use. In our calculator example we really dont have such implementation but in a general sense, we should have a class which only has one function or method that performs a single task and that class is inherited by multiple classes. For example, we can have a class that prints the result but with one change that it always doubles the result. Now that class can be called for any four or all four calculation class and when we use that class method to print the result, the result will always be double of the actual result.

Dependency Inversion Principle

Perhaps the simplest of the principles, this states that classes should depend upon abstractions, not concretions. Explicit is better than Implicit. In our case, the parent class "Calculations.py" is not dependent on its child class which is "Addition.py" but rather it relies on the abstract implementation of the child class's get_result() method. The parent class is not dependent on how the get_result() is implemented in the child class.

What is OOP?
Object-oriented programming combines a group of data attributes with functions or methods into a unit called an "object." Typically, OOP languages are class-based, which means that a class defines the data attributes and functions as a blueprint for creating objects, which are instances of the class.
What are four basics of OOP?
Object-oriented programming has four basic concepts:
  • Encapsulation
  • Abstraction
  • Inheritance
  • Polymorphism
Encapsulation?
The word, “encapsulate,” means to enclose something. Just like a pill "encapsulates" or contains the medication inside of its coating, the principle of encapsulation works in a similar way in OOP: by forming a protective barrier around the information contained within a class from the rest of the code.
Abstraction?
Abstracting away the gory implementation details of a class and only presenting a clean and easy-to-use interface via the class’ member functions. Carefully used, abstraction helps isolate the impact of changes made to the code, so that if something goes wrong, the change will only affect the implementation details of a class and not the outside code.
Inheritance?
Object-oriented languages that support classes almost always support the notion of “inheritance.” Classes can be organized into hierarchies, where a class might have one or more parent or child classes. If a class has a parent class, we say it is derived or inherited from the parent class and it represents an “IS-A” type relationship. That is to say, the child class “IS-A” type of the parent class..
Polymorphism?
In OOP, Polymorphism allows for the uniform treatment of classes in a hierarchy. Therefore, calling code only needs to be written to handle objects from the root of the hierarchy, and any object instantiated by any child class in the hierarchy will be handled in the same way.
